# DAgent 多会话与设备控制 — 技术实现方案

## 一、现状分析

**当前架构的核心瓶颈：**

- **单任务单设备**：`AgentLoop` 内部所有状态（`agent_state`、`agent_stats`、`memory`、`state_machine`）都是实例变量，一个 `AgentLoop` 实例只能串行执行一个 `Task`，无法并发
- **设备绑定死**：`AgentLoop.__init__` 中一次性创建 `device`/`perception`/`executor`，整个生命周期共享同一台设备
- **无会话隔离**：`TaskManager` 维护了一个 `tasks: dict`，但执行时全部委托给同一个 `AgentLoop`，没有会话（Session）概念
- **无设备池**：没有设备发现、注册、分配、互斥机制

## 二、核心设计目标

| 目标 | 说明 |
|------|------|
| **多会话并发** | 多个任务可同时在不同设备上并行执行，各自独立的状态/记忆/日志 |
| **设备池管理** | 统一注册、发现、分配、回收设备，支持 iOS/Android/Mock 混合 |
| **会话隔离** | 每个 Session 拥有独立的 `AgentLoop` 实例、`Memory`、`artifact_dir`，互不干扰 |
| **设备互斥** | 同一台设备同一时刻只能被一个 Session 占用（除非显式共享） |
| **最小侵入** | 不修改现有模块（`Planner`、`Executor`、`Validator` 等）的内部逻辑 |

### 3.2 关键模块

#### 模块 1：`Session` — 会话容器

每个 `Session` 是一个完全独立的执行环境：

```python
@dataclass
class Session:
    id: str                          # 会话唯一标识
    device: Device                   # 绑定的设备实例
    agent_loop: AgentLoop            # 独立的 AgentLoop 实例
    memory: Memory                   # 独立的记忆
    task: Optional[Task] = None      # 当前执行的任务
    status: SessionStatus = CREATED  # 会话状态
    created_at: float = 0.0
    artifact_dir: Path = None        # 独立的产物目录

class SessionStatus(Enum):
    CREATED = "created"
    RUNNING = "running"
    IDLE = "idle"        # 设备已绑定但无任务
    TERMINATED = "terminated"
```

**关键设计**：每个 Session 创建时实例化全新的 `AgentLoop`，注入独立的 `Device`、`Memory`、`PerceptionLayer`、`Executor`，确保状态完全隔离。

#### 模块 2：`DevicePool` — 设备池

```python
class DevicePool:
    """设备注册中心，管理所有可用设备的生命周期。"""

    def __init__(self):
        self._devices: dict[str, DeviceEntry] = {}  # device_id → DeviceEntry
        self._lock = threading.Lock()

    def register(self, device: Device, device_id: str, tags: list[str] = None) -> None:
        """注册设备到池中。"""

    def unregister(self, device_id: str) -> None:
        """移除设备。"""

    def acquire(self, requirements: DeviceRequirements = None) -> Optional[Device]:
        """按条件获取一台空闲设备（加锁，互斥）。"""

    def release(self, device_id: str) -> None:
        """归还设备（解锁）。"""

    def health_check_all(self) -> dict[str, DeviceStatus]:
        """批量健康检查。"""

    def list_available(self, requirements: DeviceRequirements = None) -> list[DeviceEntry]:
        """列出满足条件的可用设备。"""

@dataclass
class DeviceEntry:
    device: Device
    device_id: str
    platform: PlatformType
    tags: list[str]
    status: DeviceAllocationStatus  # AVAILABLE / IN_USE / OFFLINE
    locked_by: Optional[str] = None  # session_id
    last_health_check: float = 0.0

@dataclass
class DeviceRequirements:
    platform: Optional[PlatformType] = None
    tags: list[str] = field(default_factory=list)
    min_screen_width: int = 0
```

**互斥机制**：`acquire()` 使用 `threading.Lock` 保证同一设备只能被一个 Session 占用。获取时标记 `IN_USE`，`release()` 时恢复 `AVAILABLE`。

#### 模块 3：`SessionManager` — 会话管理器

```python
class SessionManager:
    """管理所有会话的生命周期。"""

    def __init__(self, device_pool: DevicePool, artifact_root: str = ".artifacts"):
        self.device_pool = device_pool
        self.artifact_root = Path(artifact_root)
        self._sessions: dict[str, Session] = {}

    def create_session(
        self,
        device_requirements: DeviceRequirements = None,
        session_id: str = None,
    ) -> Session:
        """
        1. 从 DevicePool 获取设备
        2. 创建独立的 Memory、AgentLoop
        3. 绑定到 Session
        """

    def submit_task(self, session_id: str, task: Task) -> bool:
        """向指定会话提交任务，异步执行。"""

    def get_session(self, session_id: str) -> Optional[Session]:
        """查询会话状态。"""

    def terminate_session(self, session_id: str) -> None:
        """终止会话，释放设备。"""

    def list_sessions(self) -> list[Session]:
        """列出所有会话。"""
```

## 四、实现步骤

### Phase 1：基础设施（设备池 + 会话容器）

**涉及文件**：新建 `agent/session.py`、`agent/device_pool.py`

```
1. 新建 agent/device_pool.py
   → 实现 DevicePool、DeviceEntry、DeviceRequirements
   → 验证：单测 register/acquire/release 流程，验证互斥

2. 新建 agent/session.py
   → 实现 Session、SessionStatus、SessionManager
   → create_session() 内部创建独立的 AgentLoop 实例
   → 验证：两个 Session 使用不同 MockDevice，状态互不影响
```

**`SessionManager.create_session` 核心逻辑**：

```python
def create_session(self, device_requirements=None, session_id=None):
    sid = session_id or uuid.uuid4().hex[:12]

    # 1. 从设备池获取设备
    device = self.device_pool.acquire(device_requirements)
    if not device:
        raise NoAvailableDeviceError(...)

    # 2. 创建独立的组件实例
    memory = Memory()
    perception = PerceptionLayer(device)
    executor = Executor(device)

    # 3. 创建独立的 artifact 目录
    artifact_dir = self.artifact_root / f"session_{sid}"
    artifact_dir.mkdir(parents=True, exist_ok=True)

    # 4. 创建独立的 AgentLoop（关键：新实例，不共享状态）
    agent_loop = AgentLoop(
        perception=perception,
        executor=executor,
        memory=memory,
        artifact_root=artifact_dir,
    )

    session = Session(
        id=sid,
        device=device,
        agent_loop=agent_loop,
        memory=memory,
        artifact_dir=artifact_dir,
    )
    self._sessions[sid] = session
    return session
```

### Phase 2：并发执行

**涉及文件**：`agent/session.py`（扩展）

```
3. submit_task() 使用线程池异步执行
   → 每个 Session 的任务在独立线程中运行
   → 验证：两个 Session 并发执行不同任务，日志/截图/状态各自隔离

4. 添加设备健康检查守护线程
   → 定期检查所有 IN_USE 设备的连接状态
   → 设备掉线时触发对应 Session 的 RECOVER 流程
```

**并发执行核心逻辑**：

```python
class SessionManager:
    def __init__(self, ...):
        self._executor = ThreadPoolExecutor(max_workers=8)  # 最多 8 个并发会话
        self._futures: dict[str, Future] = {}

    def submit_task(self, session_id: str, task: Task) -> bool:
        session = self._sessions[session_id]
        if session.status == SessionStatus.RUNNING:
            raise SessionBusyError(f"Session {session_id} 正在执行任务")

        session.status = SessionStatus.RUNNING
        session.task = task

        future = self._executor.submit(self._run_task, session, task)
        self._futures[session_id] = future
        return True

    def _run_task(self, session: Session, task: Task) -> bool:
        try:
            # 设置设备的 artifact 目录（截图隔离）
            if hasattr(session.device, 'set_artifact_dir'):
                session.device.set_artifact_dir(session.artifact_dir)
            return session.agent_loop.run_task(task)
        finally:
            session.status = SessionStatus.IDLE
            session.task = None
```

## 五、数据流示意

### 5.1 创建会话并提交任务

```
User → POST /sessions { requirements: { platform: "ios" } }
         │
         ▼
SessionManager.create_session()
         │
         ├── DevicePool.acquire(platform=IOS)
         │         └── 返回 iPhone 15（标记 IN_USE）
         │
         ├── new Memory()
         ├── new PerceptionLayer(iPhone15)
         ├── new Executor(iPhone15)
         ├── new AgentLoop(perception, executor, memory)
         │
         └── 返回 Session(id="a1b2c3", device=iPhone15)

User → POST /sessions/a1b2c3/tasks { goal: "打开微信搜索xxx" }
         │
         ▼
SessionManager.submit_task("a1b2c3", task)
         │
         └── ThreadPoolExecutor → agent_loop.run_task(task)
                    │
                    └── observe → decide → execute → validate（独立循环）
```

### 5.2 多会话并行

```
Session A (iPhone 15)  ──→ "搜索 Boss 直聘"    ──→ 线程 1
Session B (Pixel 8)    ──→ "打开美团点外卖"      ──→ 线程 2
Session C (MockDevice) ──→ "测试登录流程"        ──→ 线程 3
```

三个会话完全独立：各自的 `Memory`、`Observation`、截图目录、事件日志互不干扰。

## 六、关键设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| **并发模型** | `ThreadPoolExecutor` | Agent 是 I/O 密集型（等 LLM 响应 + 等设备操作），线程足够，无需协程 |
| **状态隔离** | 每个 Session 新建 `AgentLoop` 实例 | 避免共享状态导致的竞态问题，改动最小 |
| **设备互斥** | `threading.Lock` + 标记位 | 简单可靠，不需要分布式锁 |
| **截图隔离** | 每个 Session 独立 `artifact_dir` | 现有 `IOSWDADevice.set_artifact_dir()` 已支持 |
| **Planner/LLM** | 多个 Session 共享同一个 `LLMClient` | LLM 是无状态的 HTTP 调用，可安全共享 |

## 七、对现有代码的改动范围

| 文件 | 改动 | 说明 |
|------|------|------|
| `agent/loop.py` | **不改** | `AgentLoop` 本身不感知多会话，每个 Session 创建独立实例即可 |
| `agent/task_manager.py` | **不改或轻改** | 可由 `SessionManager` 替代其职责 |
| `agent/device/device.py` | **不改** | 抽象接口不变 |
| `ios/ios_device.py` | **不改** | 已有 `set_artifact_dir()` 支持目录切换 |
| `agent/device_pool.py` | **新建** | 设备注册/分配/回收 |
| `agent/session.py` | **新建** | 会话容器 + 会话管理器 |
| `agent/models.py` | **小改** | 添加 `SessionStatus`、`DeviceAllocationStatus` 枚举 |

**核心原则**：现有模块（`Planner`、`Executor`、`Validator`、`Memory`、`PerceptionLayer`）全部不修改，通过组合方式在 `Session` 层实现隔离。

## 八、风险与注意事项

- **LLM 并发限流**：多个 Session 同时调用 LLM 可能触发 rate limit，建议在 `LLMClient` 中加入重试和限流
- **设备掉线恢复**：守护线程检测到设备掉线后，应暂停对应 Session 并触发 `RecoveryManager`
- **资源清理**：`terminate_session()` 时必须调用 `device_pool.release()` 归还设备，避免设备泄露
- **日志聚合**：各 Session 的 `events.jsonl` 在各自 `artifact_dir` 下，如需全局视图需要一个聚合层
